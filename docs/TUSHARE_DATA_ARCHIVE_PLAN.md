# Tushare A 股长期数据归档计划

本文用于在 Tushare 高积分账号有效期内，建立一份可长期离线使用、可校验、可追溯
的 A 股研究数据资产。它与 `TUSHARE_DATA_REQUIREMENTS.md` 的关系如下：

- `TUSHARE_DATA_REQUIREMENTS.md`：当前 V2 Alpha2/Alpha3 的最小可运行闭环。
- 本文：面向后续 Alpha、风险、组合、成交、事件、行业和多资产研究的长期归档。

如果数据并非来自本人官方 Tushare Token，而是由兼容网关或第三方代理提供，必须
同时遵循 `DATA_ACQUISITION_V2_DESIGN.md` 的授权、隔离、原始留存和交叉校验要求。

目标是“尽可能完整地保存有研究价值的原始数据”，不是盲目下载 Tushare 的每一个
接口。分钟/Tick、新闻全文、海外市场等独立权限或超大数据，只有明确研究路线时才应
购买和归档。

## 1. 总体结论

### 1.1 账号是否足够

- 15000 积分账号足以高效下载绝大多数常规 A 股日频、财务、特色和研究数据。
- 15000 积分不等于拥有所有数据。历史分钟、实时行情、新闻、公告全文等通常是独立
  权限，应在下载前用实际 Token 做接口探针并生成权限报告。
- 本项目的可信日频回测不依赖分钟数据。日线不是“所有未来研究都够”，但足以覆盖
  当前选股、PIT 基本面、组合构建和日频成交模拟的主路线。

### 1.2 归档边界

- 市场：上海、深圳、北京证券交易所全部 A 股，包括上市、退市、暂停上市证券。
- 时间：每个接口从其最早可用日期下载到账号到期前最后一个完整交易日。
- 财务：尽量从 1990 年代/接口最早期开始；至少完整覆盖 2000 年以后。
- 当前正式研究仍只使用预先冻结的回测区间。多下载的数据不能倒流进已经看过结果的
  保留期调参。
- 原始层保存接口返回的**全部字段**，不要只保存当前代码使用的列。

## 2. 数据分层与不可变规则

建议目录结构：

```text
data_lake/
  bronze/tushare/<endpoint>/<snapshot_id>/...   # 原始响应，不覆盖
  silver/<dataset>/...                          # 清洗、统一类型、去重
  gold/qf/...                                   # 项目当前可直接读取的 PIT 缓存
  manifests/<snapshot_id>.json
  schemas/<endpoint>/<schema_version>.json
```

每次请求必须记录：

- 接口名、请求参数、起止日期、分页/分段键；
- UTC 下载时间、Tushare SDK 版本、归档程序 Git commit；
- 返回列名与数据类型、行数、最小/最大日期、证券数量；
- 单文件 SHA256、整批 Merkle/汇总 SHA256；
- 成功、空结果、重试、权限不足和截断状态；
- 账户权限探针结果，但绝不能记录 Token。

硬性规则：

1. Bronze 原始层只追加，不覆盖；上游修订后产生新 `snapshot_id`。
2. 原始层使用 Parquet（推荐 Zstandard 压缩），同时保留请求 manifest。
3. 股票代码、公告日、报告期、报告类型、修订标记和有效区间不得在清洗时丢失。
4. 任何达到接口单次行数上限的请求都视为“疑似截断”，必须继续拆分重取。
5. 原始数据不进入 Git；至少保留本机和另一块磁盘/对象存储两份，并校验 SHA256。

## 3. P0：必须拿齐——任何可信 A 股研究的底座

P0 下载并通过验收后，才算没有阻塞当前 V2 和大多数后续日频研究。

### 3.1 证券身份、交易日历与可交易状态

- [ ] `trade_cal`：SSE、SZSE、BSE（接口支持范围内），全历史开放/关闭日期。
- [ ] `stock_basic`：分别获取 `L/D/P`，保留接口全部字段。
- [ ] 股票历史列表、上市公司基本信息 `stock_company`。
- [ ] `namechange`：全市场、全历史，用于 ST/*ST、改名和证券身份时段。
- [ ] `new_share`：IPO 日期、发行价、发行量等。
- [ ] 历史 ST/风险警示清单（若当前账号接口可用）。
- [ ] `suspend_d`：停复牌类型、时间和原因。
- [ ] `stk_limit`：每日涨跌停价。

验收重点：退市股不能缺失；不能以今天的股票名单回填历史；股票身份必须能按日期重建。

### 3.2 全市场日频行情与复权

- [ ] `daily`：全部 A 股、接口最早日期至今，保存全部字段。
- [ ] `adj_factor`：全部 A 股、全历史。
- [ ] `daily_basic`：全部 A 股、全历史、全部字段。
- [ ] `dividend`：全部方案和状态，保留实施/预案/取消记录。
- [ ] `block_trade`：大宗交易，全历史。

建议按 `trade_date` 分区拉取全市场数据，再按年写 Parquet；不要对 5000 多只股票逐只
制造大量小文件。复权价格应在 Silver/Gold 层动态计算，Bronze 同时保留原始价格和
复权因子。

### 3.3 指数、历史成分与行业分类

- [ ] `index_basic`：主要市场和指数公司全部指数元数据。
- [ ] `index_daily`：至少沪深 300、中证 500、中证 1000、上证 50、科创 50、创业板指
  的价格指数和可取得的全收益指数；建议保存全部主要宽基。
- [ ] `index_weight`：上述指数全部历史权重，不只下载当前成分。
- [ ] 申万 2014/2021 行业的 `index_classify` 与 `index_member_all`，L1/L2/L3 全层级。
- [ ] 可取得的申万/中信行业指数日线及估值。

历史成分和行业必须保存 `in_date/out_date` 或等价有效区间。只有当前成员、没有有效期
的数据可以归档，但不得伪装成 PIT 历史成员数据。

### 3.4 PIT 财务报表——全部字段、全部版本

- [ ] `income` / `income_vip`
- [ ] `balancesheet` / `balancesheet_vip`
- [ ] `cashflow` / `cashflow_vip`
- [ ] `fina_indicator` / `fina_indicator_vip`
- [ ] `forecast` / `forecast_vip`
- [ ] `express` / `express_vip`
- [ ] `fina_audit`
- [ ] `fina_mainbz` / `fina_mainbz_vip`：按产品、地区、行业全部下载。
- [ ] `disclosure_date`

归档要求：

- 保存全部返回字段，而非当前因子所需的十几个字段。
- 保存所有 `report_type/comp_type/update_flag`、公告日、实际公告日、报告期和修订版本。
- VIP 接口适合按季度批量下载，但仍要做逐季行数上限检查，并抽样与单股票接口核对。
- 不要在 Bronze 层去重为“每个报告期一条”；更正公告和历史修订正是 PIT 研究需要的
  信息。

## 4. P1：强烈建议拿齐——未来最可能产生增量价值

P1 数据多为 Tushare 整理后的事件或特色数据，体量相对可控，账号失效后重新补齐更
麻烦。P0 完成后应优先下载。

### 4.1 股本、股东与公司治理事件

- [ ] `repurchase`：股份回购。
- [ ] `share_float`：限售股解禁。
- [ ] `pledge_stat`、`pledge_detail`：股权质押。
- [ ] `stk_holdernumber`：股东户数。
- [ ] `stk_holdertrade`：重要股东增减持。
- [ ] `top10_holders`、`top10_floatholders`：十大股东/流通股东。
- [ ] 高管名单与薪酬、管理层变动等公司治理数据。

这些数据适合事件研究、拥挤度、治理风险、供给冲击和基本面质量因子。

### 4.2 资金、杠杆、异常交易与市场微观结构（日频）

- [ ] `moneyflow`：个股资金流，全历史。
- [ ] `margin`、`margin_detail`、`margin_secs`：两融汇总、个股明细和标的范围。
- [ ] 沪深港通资金/持股数据（能取得的完整历史）。
- [ ] `top_list`、`top_inst`：龙虎榜与机构明细。
- [ ] `limit_list_d`、`limit_list_ths`：涨跌停统计和连板特征。
- [ ] 集合竞价日级汇总（如账号已有权限且能历史回补）；不为此单独购买实时权限。

### 4.3 分析师、机构预期与调研

- [ ] `report_rc`：卖方盈利预测与估值预测。
- [ ] `stk_surv`：机构调研活动。
- [ ] `broker_recommend`：券商月度金股/推荐。

这些是盈利预期修正、关注度和情绪研究的重要原始输入。必须保留报告发布日期、预测
年度、机构和分析师身份，不能只保留最新一致预期。

### 4.4 筹码数据

- [ ] `cyq_perf`：获利比例、成本分位和集中度；优先下载，体量相对可控。
- [ ] `cyq_chips`：逐价格档筹码分布；先抽取 1 个月估算总行数和磁盘，再决定全量。

`cyq_chips` 可能是本计划中体量最大的日频特色表之一。若磁盘或时间不足，保留
`cyq_perf`，把 `cyq_chips` 降为 P2。

### 4.5 ETF、基金与机构持仓

- [ ] 场内 ETF/LOF 基本信息、日线、复权因子、份额和净值。
- [ ] 公募基金基本信息、净值、分红、规模、资产配置和股票持仓披露。
- [ ] 主要宽基 ETF 与指数成分建立可核对映射。

用途：资金流代理、机构拥挤、风格轮动、对冲基准和实盘可交易替代品。至少把 ETF
全量拿齐；普通公募基金持仓可按数据量分批。

### 4.6 低体量宏观与跨资产状态变量

- [ ] Shibor、LPR、国债收益率曲线。
- [ ] GDP、CPI、PPI、PMI、货币供应量、社会融资规模。
- [ ] 股指期货 `IF/IH/IC/IM` 合约信息与日线。
- [ ] 主要股指期权/ETF 期权合约信息与日线。

这些表体量小，适合市场状态、风险预算、流动性和对冲研究，建议一次性封存。

## 5. P2：有空间再拿——大体量、重复或主要可重算

- [ ] `stk_factor_pro`：Tushare 预计算技术因子全历史。它便于对照和快速研究，但绝大
  多数可由 OHLCV 重算；若下载，优先 Pro 版本，不必同时全量保存旧 `stk_factor`。
- [ ] `cyq_chips` 全量：只有在抽样估算通过磁盘/时间预算后执行。
- [ ] `moneyflow_ths`、`moneyflow_dc`：与基础 `moneyflow` 来源/口径不同，可用于
  供应商一致性研究，但存在重复。
- [ ] 同花顺、东方财富、通达信概念/板块元数据、成分和日线。
- [ ] 主题、热点榜、连板榜等短历史供应商衍生表。
- [ ] 全部商品期货、期权和可转债数据；仅在确定扩展为多资产/转债策略时全量下载。

概念板块若没有可靠的历史纳入/剔除日期，只能按下载日保存快照。日后研究必须明确
标注“非 PIT”，不能用今天的概念成员回测过去。

## 6. P3：本项目当前不应花时间或额外付费的数据

这里的“跳过”是针对“A 股日频 PIT 量化研究”的投入产出，不代表数据在所有场景
绝对无用。

- 周线、月线：可由完整日线确定性聚合，不必重复归档。
- `pro_bar` 生成的前/后复权 K 线：可由 `daily + adj_factor` 动态生成；只做抽样核对。
- 实时快照、实时 Tick、实时分钟：无法回补历史，且不是当前研究所需。
- 历史分钟/Tick/Level-2：体量和独立权限成本很高；只有开发日内/高频成交模型时再买。
- 新闻全文、公告 PDF、互动问答全文：独立权限、文本体量大；只有明确 NLP 路线时下载。
- 港股、美股、外汇、数字货币、商品现货等全市场历史：没有跨市场路线时不归档。
- 黑箱择时指标、重复行情备份、仅展示用途榜单：可解释性低或能从底层数据重算。
- 已停止维护且有正式替代接口的旧表：保存替代接口，不为旧表做全量重复下载。

## 7. 账号有效期内的下载顺序

按以下顺序执行，不要一开始就跑最大的筹码或技术因子表：

1. **权限探针与 schema 快照**：对清单中每个接口取 1 个交易日/1 个季度，记录权限、
   列名、单次上限和最早可用日期。
2. **P0 身份、日线、复权、交易状态和指数成员**：先保证完整股票历史与价格底座。
3. **P0 财务全版本**：按季度批量，做截断检测和抽样交叉核验。
4. **P1 难替代特色数据**：分析师、调研、筹码绩效、股东事件、两融、龙虎榜等。
5. **P1 ETF/基金、宏观和股指衍生品**。
6. **P2 大表**：先做小样本容量外推，设置磁盘和最大运行时间阈值后再全量执行。
7. **最终增量同步**：账号到期前 7 天、前 1 天各补一次，并生成最终冻结快照。

每完成一个 endpoint 就独立封存和验收。不要等所有下载结束后才写 manifest，否则一次
中断会让已完成数据无法判断完整性。

## 8. 容量规划（仅作预算，必须以抽样实测为准）

- P0 + P1 的紧凑 Parquet 通常应按 **数十 GB** 预留。
- 加入全量预计算技术因子、详细筹码、基金持仓和多资产后，应按 **100--300 GB 或
  更高**预留。
- `cyq_chips`、宽表技术因子和文本/分钟数据会显著放大容量，禁止未抽样估算就全跑。
- 建议准备至少 500 GB 可用 SSD 作为工作盘，并准备同等或更大的第二份备份空间。

以上不是官方数据量承诺。归档程序应先下载代表性月份，按
`总交易日 × 日均行数 × Parquet 每行字节`计算实际预算。

## 9. 强制验收标准

### 9.1 每个接口

- [ ] 请求没有触达单次行数上限；若触达，已递归拆分到不触达。
- [ ] manifest 中的预期分区数等于成功分区数 + 已确认空分区数。
- [ ] 日期、证券数、行数、重复主键数、空值率和 SHA256 已记录。
- [ ] 随机抽取至少 20 个证券/日期，与原接口重取结果一致。
- [ ] 列名变化被记录为 schema 版本，而非静默丢列。

### 9.2 跨表一致性

- [ ] `daily` 中代码均能关联证券身份及上市区间。
- [ ] 交易日期属于交易日历；停牌日缺行情有可解释记录。
- [ ] 复权因子变化能与分红送转事件核对。
- [ ] 指数成分权重按日期有效，不能出现当前成分回填历史。
- [ ] 财务数据在公告日前不可见；修订版本没有被提前使用。
- [ ] 退市股、历史 ST 股和北交所证券的覆盖率单独报告。

### 9.3 交付物

- [ ] `permission_report.json`
- [ ] `endpoint_inventory.csv`
- [ ] `download_manifest.jsonl`
- [ ] `schema_registry/`
- [ ] `checksums.sha256`
- [ ] `coverage_report.html` 或等价 CSV/Markdown 报告
- [ ] Bronze 数据目录和独立备份的校验结果
- [ ] 由 Bronze 生成当前 QF Gold 缓存的固定命令与 Git commit

## 10. 给本地 Agent 的执行原则

1. 先实现通用归档器，再写 endpoint 清单；不要为每个表手工复制一套不可恢复脚本。
2. endpoint 配置应声明：日期键、主键、切分方式、单次上限、最早日期、优先级和权限。
3. 支持断点续传、幂等重跑、指数退避、限速和明确的失败队列。
4. 空响应必须区分“真实无数据”“权限不足”“参数错误”“接口超时”。
5. 下载完成后从本地 Bronze 构建研究缓存，正式回测不得在线临时补数据。
6. Token 到期前先冻结最后快照，再撤销环境变量并扫描仓库，确认 Token 未泄露。

## 11. 最终拍板

本项目建议采用以下采购/下载决策：

- **必须全量**：P0 全部。
- **账号有效期内尽量全量**：P1 全部，其中 `cyq_chips` 先做容量评估。
- **有空间再全量**：P2；优先 `stk_factor_pro` 和概念/资金流异源数据。
- **不额外购买**：P3 中的分钟、Tick、Level-2、实时和全文数据，除非后续正式立项。

这样可以覆盖未来数年的 A 股日频选股、基本面、事件、情绪、行业轮动、风险状态、
组合与日频成交研究，同时避免为可从底层数据重算或与研究路线无关的数据付出最高的
存储和权限成本。

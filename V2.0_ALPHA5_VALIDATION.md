# V2.0 Alpha5 验证记录

版本：`2.0.0a11`

## 1. 工程验收结论

Alpha5 新增真实 PIT 数据验收编排和封存回执，生产策略、离线 demo 的冻结数值以及 Alpha2/Alpha3 因子公式均未改变。合成 strict 归档、fixture 归档和故障注入测试已通过；真实全量结论必须等待下载侧 B1 隔离任务闭环且 B3 完成。

## 2. 自动化检查

在干净环境执行：

```bash
uv lock --check --default-index https://pypi.org/simple
uv sync --frozen --extra public --default-index https://pypi.org/simple
uv run --no-sync ruff check src tests
uv run --no-sync python -W error::FutureWarning \
  -m unittest discover -s tests -v
```

预期结果：

- Ruff 无问题；
- 225/225 项测试通过；
- 未安装真实 MiniRacer 运行时时，两个可选压力测试允许显示为 skip；
- PIT lake 与 Alpha5 相关测试在 `DeprecationWarning` 升级为错误时也必须通过。

## 3. Alpha5 专项覆盖

| 场景 | 预期 |
| --- | --- |
| strict 合成归档完整 | `decision=pass`，回执与 PIT 指纹绑定 |
| fixture 合成归档完整 | `decision=engineering_only`，禁止研究 |
| B3 决策证据缺失 | 写出可封存的 `blocked` 报告 |
| 缓存构建后修改 Bronze | 源重放失败，复用缓存也被阻断 |
| 历史估值超过允许陈旧天数 | 对应月末进入 `dates_below_threshold` |
| 回执正文或 manifest 指纹被修改 | 下游校验拒绝 |
| strict manifest 把门禁标志改为 false | 下游校验拒绝 |
| 验收输出目录非空 | 拒绝覆盖或混入旧产物 |
| 批次含 `quarantined` 任务 | `no_quarantined=false`，批次不得 pass |
| macOS `/tmp` 映射为 `/private/tmp` | 路径解析后测试通过，不改变业务逻辑 |

## 4. 命令行检查

```bash
python run.py pit-acceptance --help
python run.py pit-research --help
python run.py pit-shadow --help
```

`pit-research` 与 `pit-shadow` 都应显示 `--acceptance-report`。`pit-acceptance` 在 `blocked` 时返回退出码 2，方便 CI 或本地 Agent 失败即停。

## 5. fixture 工程验收

```bash
python run.py pit-acceptance \
  --config config.yaml \
  --archive-root /absolute/path/to/QF-fixtures \
  --fixture-mode \
  --output results/pit_acceptance_fixture

python run.py result-verify \
  --output results/pit_acceptance_fixture --strict
```

预期：命令成功，四个报告文件通过 SHA256 封存，但 `decision=engineering_only`、`research_eligible=false`。这只证明工程管线，不是全量数据验收。

## 6. 真实归档最终验收清单

下载完成后，本地 Agent 应在活动下载进程停止并固定快照后执行：

1. B0、B1、B3 的 `batch_decision.json` 都是同一快照且 `decision=pass`。
2. 任一批次的 `quarantined`、`suspect_truncated`、`retryable_error`、`denied` 和 `invalid_params` 都为 0。
3. 基础行情 `manifest.json`、`membership.csv.gz` 和归档任务覆盖相同请求区间。
4. 在新的空输出目录运行 `pit-acceptance`，不使用 `--fixture-mode`。
5. `acceptance_report.json` 必须为 `mode=strict`、`decision=pass`。
6. `source_files_verified == selected_task_count`。
7. 所有月末时点的财报与估值覆盖率达到配置阈值。
8. `result-verify --strict` 通过，未封存文件数为 0。
9. 再运行 Alpha2/Alpha3，并显式传入该验收回执。

建议保存以下身份用于跨机器比较：

- ZIP SHA256；
- Git commit SHA；
- 基础行情数据指纹；
- PIT 数据指纹；
- selected task set SHA256；
- Alpha5 acceptance fingerprint；
- Alpha2/Alpha3 artifact set fingerprint。

## 7. 当前限制

- 云端没有用户本地全量 data lake，因此不能替代本地完成真实 `pass` 验收。
- fixture 中的证券数和历史长度不足以评价收益或全市场覆盖。
- 公开数据源或后复权因子会漂移；公开回测不能替代绑定归档字节的 strict PIT 验收。
- Alpha5 只保证数据与运行证据可信，不代表候选因子有效，更不构成达到 15% 年化的保证。

## 8. 回归边界

离线 demo 应与 v1.6 冻结基线一致。若净值、订单、成交、选择、指标或费用的 SHA256 变化，应视为非预期业务回归并停止发布，而不能只比较最终年化数字。

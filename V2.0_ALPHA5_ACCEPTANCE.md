# V2.0 Alpha5：真实 PIT 数据验收与研究准入门禁

版本：`2.0.0a11`

## 1. 目标

Alpha4 证明归档中的 Bronze/Schema/批次证据可以确定性转换为 PIT 缓存；Alpha5 解决另一个问题：**什么条件下，这份缓存才可以进入 Alpha2/Alpha3 研究**。

新命令 `pit-acceptance` 把缓存构建或复用、源证据重放、PIT 时点规则、历史股票池覆盖和结果封存串成单一失败即关闭的流程。它不修改生产策略、不访问网络，也不向正在运行的下载目录写入数据。

## 2. 决策状态

| `decision` | 含义 | 可运行真实 Alpha 研究 |
| --- | --- | --- |
| `pass` | strict 模式全部阶段通过，且 PIT 缓存本身具备研究资格 | 是，仅限与回执完全相同的数据指纹 |
| `engineering_only` | fixture 工程样例通过，但 manifest 固定 `research_eligible=false` | 否 |
| `blocked` | 任一必要阶段失败、缺证据或覆盖不足 | 否；命令返回退出码 2 |

Alpha4 的 `research_eligible=true` 只是必要条件。对新 strict 缓存，Alpha5 的 `pass` 回执才是最终的下游准入凭证。

## 3. 一键验收

在 B0、B1、B3 均完成且批次决策为 `pass` 后执行：

```bash
python run.py pit-acceptance \
  --config config.yaml \
  --archive-root /absolute/path/to/data_lake \
  --output results/pit_acceptance_v2_alpha5

python run.py result-verify \
  --output results/pit_acceptance_v2_alpha5 --strict
```

默认复用 `point_in_time.cache_dir` 中的现有缓存；缓存不存在时自动构建。需要明确重建时使用 `--rebuild-cache`。输出目录必须不存在或为空，程序不会把本次结果混入旧报告。

若归档状态库、Schema 或报告不在标准位置，可显式传入：

```bash
python run.py pit-acceptance \
  --config config.yaml \
  --archive-root /absolute/path/to/data_lake \
  --catalog /absolute/path/to/archive.duckdb \
  --schema-registry /absolute/path/to/schemas \
  --reports-root /absolute/path/to/reports \
  --cache /absolute/path/to/pit_cache \
  --buckets 32 \
  --output results/pit_acceptance_v2_alpha5
```

## 4. 验收阶段

### 4.1 缓存构建或复用

- 缓存不存在时调用 Alpha4 离线桥接构建。
- 已存在时复用并在后续阶段重新验证，不能只信旧结论。
- `--rebuild-cache` 采用原子替换语义。
- 报告记录 `built`、`rebuilt` 或 `reused` 以及 PIT 数据指纹。

### 4.2 完整源证据重放

- 状态库只读打开。
- B0/B1/B3 必须来自同一快照并全部 `pass`。
- 批次任务集合、基础行情身份、Schema、Parquet 行数与 Bronze SHA256 全部复核。
- `source_files_verified` 必须与选中任务数一致。
- 源文件在缓存构建后被修改时，复用缓存也会被阻断。

批次决策新增硬门 `no_quarantined`。`quarantined` 对下载状态机可以是终态，但绝不是研究可接受终态；即使成功率高于 99.5%，一个隔离任务也不能被平均成功率掩盖。

### 4.3 PIT 时点与历史覆盖

程序在回测区间内取每月最后一个交易日，并用该时点最近且不超过 62 天的历史成分快照作为分母，检查：

- 财报只在公告及配置滞后之后可见；后续修订不能改写过去快照。
- 估值只使用当日及以前的记录。
- 财报和估值不能超过 manifest 请求结束日。
- 估值键重复、财报修订序列异常和滞后规则不一致均会失败。
- 每个时点分别计算财报和估值覆盖率。
- 记录的最近可见日期必须满足 `maximum_fundamental_age_days` / `maximum_valuation_age_days`。
- 两类覆盖率都不得低于 `minimum_symbol_coverage`。

报告保存最低值、5% 分位数、中位数、最新值、所有未达标日期和最多 20 个缺失证券样例。这里检查的是**历史活跃股票池**，不是用今天仍存在的股票回填过去。

## 5. 防篡改回执与下游绑定

严格缓存的 `archive_bridge` 使用 schema v2，并写入 `acceptance_required=true`。Alpha2/Alpha3 在开始计算前会：

1. 严格校验验收目录的 `artifact_manifest.json`；
2. 要求回执为 strict 模式且 `decision=pass`；
3. 重新计算验收身份指纹；
4. 核对 PIT、基础行情、任务集合、批次快照和桥接版本身份；
5. 把验收回执作为研究复现输入写入自身封存信息。

因此，复制另一份数据的回执、修改回执正文、修改 PIT manifest 指纹，或把 `acceptance_required` 人为改为 `false` 都会被拒绝。Alpha4 旧版 schema v1 缓存保持可读兼容，但升级到本版本后也必须补跑 Alpha5 回执；新建 strict 缓存还会通过显式门禁标志做第二层校验。

下游命令：

```bash
python run.py pit-research \
  --config config.yaml \
  --acceptance-report results/pit_acceptance_v2_alpha5/acceptance_report.json \
  --output results/pit_factor_research_v2_alpha2

python run.py pit-shadow \
  --config config.yaml \
  --acceptance-report results/pit_acceptance_v2_alpha5/acceptance_report.json \
  --alpha2-research results/pit_factor_research_v2_alpha2 \
  --output results/pit_shadow_v2_alpha3
```

## 6. 验收产物

| 文件 | 内容 |
| --- | --- |
| `acceptance_report.json` | 完整阶段状态、源复核、时间切片覆盖、PIT 身份与决策 |
| `acceptance_report.md` | 便于人工复核的摘要 |
| `reproducibility.json` | 配置、代码、运行环境与数据 manifest 身份 |
| `artifact_manifest.json` | 上述产物逐文件 SHA256 与集合指纹 |

## 7. fixture 与真实数据边界

```bash
python run.py pit-acceptance \
  --config config.yaml \
  --archive-root /path/to/QF-fixtures \
  --fixture-mode \
  --output results/pit_acceptance_fixture
```

fixture 通过时只能得到 `engineering_only`。它可验证代码、Schema、路径重映射和封存机制，但不能证明全市场覆盖，不能生成可晋级的 Alpha 证据。

真实数据验收必须等待：

- B1 的隔离任务得到成功重跑或被有证据地确认为业务空结果；
- B3 全量任务完成且批次决策为 `pass`；
- 基础行情历史成分、基础 manifest 与 PIT 请求区间匹配。

这些条件不妨碍 Alpha5 工程代码并行开发，但在满足前，任何真实研究结论都应保持 `blocked`。

## 8. 非目标

- 不承诺任何年化收益或 Alpha 增量。
- 不自动调参，也不改变 v1.6 生产默认策略。
- 不把下载状态改写为通过，不伪造缺失分区。
- 不连接券商，不提供实盘下单。

Alpha5 的价值是让后续研究先回答“数据证据是否合格”，再回答“因子是否有效”。

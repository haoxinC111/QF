# V2.0 Alpha4：归档数据湖到 PIT 研究缓存的可信桥接

## 1. 目标与边界

Alpha4 解决的是“已经归档的数据怎样安全进入研究层”，不是新增选股因子，也不改变生产策略。`pit-lake-build` 全程离线读取 Bronze Parquet，把下载状态库、批次决策、Schema 注册表和文件 SHA256 绑定到现有 PIT v1 缓存；后续 Alpha2/Alpha3 继续消费原有 `PointInTimeDataBundle`。

本阶段不访问任何行情接口，不启动归档 worker，不写归档状态库。生产默认仍为 `legacy_v1_4 + inverse_vol_v1_4 + fixed_bps`。

## 2. 两种互斥模式

| 模式 | 用途 | 财务来源 | 准入条件 | `research_eligible` |
| --- | --- | --- | --- | --- |
| strict | 真实研究 | `income_vip`、`balancesheet_vip`、`cashflow_vip`、`fina_indicator_vip` | B0、B1、B3 全部 pass，完整批次 manifest/checksums，基础行情 v4 身份一致 | `true` |
| fixture | 跨平台工程复现 | 优先 VIP，无 VIP 时显式回退 Phase A 普通端点 | 本地样例 Parquet、状态库快照、Schema 注册表 | `false` |

fixture 模式生成的缓存可以做 `pit-snapshot`、缓存往返和血缘校验，但 `pit-research` 与 `pit-shadow` 会在读数据前硬拒绝。不能通过改命令行参数把样例升级为研究证据。

## 3. 严格证据链

严格构建逐层校验：

1. 用 SQLite `mode=ro` 打开 `catalog/archive.duckdb`，从不创建或更新状态表。
2. 读取 B0/B1/B3 的 `batch_decision.json`，要求 decision 为 pass 且所有 gate 为 true。
3. 从对应 `batch_manifest.jsonl` 选择与 decision 相同的 `snapshot_id`，要求相关任务仅为 `success` 或 `confirmed_empty`。
4. 用 `task_id` 对照当前状态库；状态、端点、行数和 Schema 指纹发生漂移即失败。
5. 把状态库里的旧机器绝对路径按 `data_lake` 锚点安全重映射到 `--archive-root`；拒绝 `..`、目录逃逸和没有锚点的外部绝对路径。
6. 每个选中 Bronze 文件按批次 `checksums.sha256` 复核 SHA256，再核对 Parquet 行数。
7. 从 `catalog/schema_registry/<api>/<fingerprint>.json` 核对端点、指纹和有序列名。不会用 Parquet 往返后的 pandas dtype 重新猜测 Schema 指纹。
8. PIT 缓存与基础行情 v4 manifest 的 SHA256、数据指纹和历史证券全集完全绑定。

B2 指数宇宙和 B4 事件批次不属于本桥接的直接输入，因此不写入 Alpha4 准入公式；实际顺序化下载链仍会先完成 B2 才启动 B3。B4 的公司行动最终由基础行情严格账本继续处理。

## 4. 转换口径

- 交易日历：B0 `trade_cal`，只使用 `is_open=1`。
- 日终估值：B1 `daily_basic`，映射为现有 `VALUATION_COLUMNS`；若某个已登记的历史源端缺陷分区没有 `trade_date`，只允许从该任务自己的 `params_json.trade_date` 补回。
- 财务报表：B3 四个 VIP 端点映射到原有 income/balancesheet/cashflow/fina_indicator 规范。
- 合并报表：income/balancesheet/cashflow 只接受 `report_type=1`；fina_indicator 按其原生无 report_type 契约处理。
- 可见日期：财报按公告/实际公告较晚者之后的第一个交易日可见；估值按配置的收盘滞后规则可见。
- 修订：保留所有源行，跨分区重新计算连续 `revision_sequence`；晚发布的修订不会重写其可见日前的历史快照。
- 冲突：同一证券同一交易日出现不一致估值时直接失败，不使用“最后一行获胜”。

## 5. 大数据处理

桥接不会把全市场 `daily_basic` 一次性拼进内存：

1. 逐个 Parquet 读取并立即过滤到基础行情历史证券全集和请求区间。
2. 用证券代码的 SHA256 稳定散列写入 32 个临时 Parquet 桶。
3. 每个桶内排序、去重和财报修订重编号。
4. 输出与原 PIT v1 兼容的每证券确定性 gzip CSV。
5. 完成整包校验后才原子发布；失败会清理专用临时目录，不触碰归档湖。

可用 `--buckets 1..128` 调整磁盘/文件句柄折中。该参数不改变最终数据语义或分区文件名。

## 6. 血缘产物

PIT 目录新增封存文件 `archive_lineage.json.gz`，内容包括：

- 选中任务的 task_id、端点、规范来源、行数、Raw SHA、Bronze SHA、Schema 指纹和相对路径；
- 选中任务集合的确定性 SHA256；
- Schema 注册文件及 SHA256；
- B0/B1/B3 decision、manifest、checksums 的路径、快照与 SHA256；
- 构建时状态库快照 SHA256、请求区间、证券全集与研究资格。

该文件本身进入 PIT 文件清单和数据指纹。`pit-lake-verify --archive-root ...` 还会重新读取当前源文件、状态库、Schema 和批次证据，对所有已选输入重放 SHA256；状态库后续新增 B4/P1 任务不会让既有选中任务失效。

## 7. 使用方式

准备好 `config.yaml`，确认 `point_in_time.enabled: true`，基础行情 v4 已通过 `validate-data`，且 B0、B1、B3 均已有完整 pass 产物，然后执行：

```bash
python run.py pit-lake-build \
  --config config.yaml \
  --archive-root data_lake

python run.py pit-lake-verify \
  --config config.yaml \
  --archive-root data_lake

python run.py pit-research \
  --config config.yaml \
  --output results/pit_factor_research_v2_alpha2
```

若状态库、Schema 或报告不在默认位置，可分别传 `--catalog`、`--schema-registry` 和 `--reports-root`。既有 PIT 目录默认不覆盖；确认是受支持的旧缓存时才加 `--force`。

工程样例只能显式执行：

```bash
python run.py pit-lake-build \
  --config config.fixture.yaml \
  --archive-root QF-fixtures \
  --catalog QF-fixtures/catalog_snapshot.sqlite \
  --schema-registry QF-integration/schemas/schema_registry \
  --fixture-mode

python run.py pit-lake-verify \
  --config config.fixture.yaml \
  --fixture-mode \
  --archive-root QF-fixtures \
  --catalog QF-fixtures/catalog_snapshot.sqlite \
  --schema-registry QF-integration/schemas/schema_registry
```

## 8. 尚未声称完成的事项

- 当前云端包只用交付样例完成了工程端到端，尚未拿到下载中的 B3 全量产物，因此没有声称完成真实 PIT 研究验收。
- Alpha4 不新增或调优因子，不支持“为了达到 15% 年化”自动寻参。
- 全量源证据重放会读取并散列所有选中 Bronze 文件，耗时受本地磁盘影响；不提供跳过严格 SHA 校验后仍标记为研究合格的捷径。
- 券商仿真、实盘下单和账户风控仍不在本项目范围内。

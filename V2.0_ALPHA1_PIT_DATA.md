# V2.0 Alpha 1：Point-in-Time 财报与估值数据基础

## 1. 阶段结论

`2.0.0a1` 只建立可审计的时点数据层，不修改 v1.6 的生产 Alpha、组合模型、成交模型或回测结果。默认配置中 `point_in_time.enabled=false`，因此旧配置和旧命令仍走冻结的 `legacy_v1_4 + inverse_vol_v1_4 + fixed_bps` 路径。

本阶段解决的是“某个历史交易日当时究竟能看到什么数据”，不是宣称基本面 Alpha 已经有效。财报/估值数据尚未进入生产选股；下一阶段必须先做因子定义、截面暴露、消融、滚动验证与晋级治理。

## 2. 数据架构

V2 使用独立侧车，不改写已冻结的 v4 行情缓存：

```text
data/cache/                 # v1.6 基础行情缓存，继续保持 v4
└── manifest.json           # 基础数据集合指纹

data/pit_cache/             # V2 PIT v1 侧车
├── manifest.json
├── calendar.csv.gz
├── fundamentals/
│   └── <symbol>.csv.gz
└── valuations/
    └── <symbol>.csv.gz
```

PIT manifest 同时保存：

- 自身所有实际读取文件的大小与 SHA256；
- 文件集合指纹 `data_fingerprint_sha256`；
- 基础行情 `manifest.json` 的 SHA256；
- 基础行情的数据集合指纹；
- 下载区间、证券全集、公告/估值滞后规则和覆盖率审计。

读取时要求“实际文件集合 = manifest 封存集合 = 程序消费集合”。增加未登记文件、修改任一分区或替换基础行情 manifest 都会拒绝加载。刷新缓存只会清理已知的 PIT 文件；目录里存在未知用户文件时会停止，避免误删。

## 3. 规范化数据契约

### 3.1 财报长表

当前 Tushare 适配器读取 `income`、`balancesheet`、`cashflow` 和 `fina_indicator`，统一转换为一行一个指标的长表。三张原始报表显式请求 `report_type=1`（最新合并报表口径），不依赖数据源默认值；财务指标接口没有该输入参数。每条记录至少包含：

| 字段 | 含义 |
| --- | --- |
| `symbol` / `source` / `metric` / `unit` | 证券、来源、稳定指标名和明确单位 |
| `period_end` | 财报期末日 |
| `announcement_date` | 保守的有效公告日 |
| `available_date` | 策略最早可使用的交易日 |
| `revision_sequence` | 同一财报期修订序号，从 1 连续递增 |
| `source_row_sha256` | 原始来源行的不可变身份 |
| `value` | 有限数值 |

当 `ann_date` 与 `f_ann_date` 同时存在时，程序使用两者中较晚者，避免把尚未正式披露的数据提前暴露。不同接口只请求各自支持的元数据字段；缺失的可选元数据规范化为空字符串，不伪造值。

### 3.2 日终估值

`daily_basic` 被规范化为交易日、可用日、换手率、PE TTM、PB、PS TTM、股息率以及总/流通市值。市值单位明确记录为“万元人民币”，不会和财报中的人民币金额混用。

## 4. 防前视规则

设公告日为 (a)，有序交易日集合为 (T)，财报滞后为 1，则：

\[
available(a)=\min\{t\in T:t>a\}
\]

也就是财报在公告当天不可见，最早从下一交易日开始可见。周末公告映射到下一交易日。默认日终估值滞后为 0，可在其交易日收盘后用于同日快照；若未来策略在开盘前生成信号，应把估值滞后改为 1。

快照算法还执行以下约束：

1. 先删除 `available_date > as_of_date` 的记录；
2. 同一报告期只保留截至当时最新可见的修订；
3. 同一指标再选择当时最新可见的报告期；
4. 按配置剔除过旧财报和估值；
5. 未来记录即使被修改，也不能改变过去快照。

严格校验会重新根据公告日、交易日历和配置滞后计算每一行的 `available_date`。手工伪造更早可用日会直接失败，而不是相信缓存中的派生字段。

## 5. 配置迁移

V2 配置结构版本为 2。没有 `schema_version` 的 v1 配置会在内存中迁移，PIT 默认关闭；未知未来版本和未知顶层字段会拒绝加载，以便尽早发现拼写错误。

```yaml
schema_version: 2

point_in_time:
  enabled: true
  provider: tushare
  cache_dir: data/pit_cache
  history_years: 6
  fundamental_lag_trading_days: 1
  valuation_lag_trading_days: 0
  maximum_fundamental_age_days: 550
  maximum_valuation_age_days: 10
  minimum_symbol_coverage: 0.90
  refresh: false
```

相对路径和基础行情缓存一样，以配置文件所在目录为基准解析。`minimum_symbol_coverage` 是整个请求区间内“至少有一条有效记录”的证券覆盖门槛，不等价于每个历史截面的可用率；后续 Alpha 研究仍须逐信号日输出覆盖率。

## 6. 使用方式

先按 v1.6 流程生成并校验基础行情，再启用 `point_in_time`：

```bash
python run.py download --config config.yaml
python run.py validate-data --config config.yaml
python run.py pit-download --config config.yaml
python run.py pit-verify --config config.yaml
```

下载按证券分别落盘，并用 `download_state.json` 记录已完成分区。中断后保持配置和基础行情不变，重跑同一命令会续传；完整封存后状态文件不会保留在正式缓存中。若身份发生变化，必须显式设置 `point_in_time.refresh=true` 重新开始。

检查某个历史日期当时可见的数据：

```bash
python run.py pit-snapshot --config config.yaml --date 2020-04-01 \
  --symbols 600000.SH,000001.SZ \
  --output results/pit_snapshot_2020-04-01.csv
```

命令同时写出 `pit_snapshot_2020-04-01.csv.manifest.json`，记录快照 SHA256、PIT 指纹、基础行情指纹、日期和请求证券。默认拒绝覆盖已有快照；确认覆盖时显式加 `--force`。省略 `--output` 时只在终端显示前 20 行。

当 `point_in_time.enabled=true` 时，普通 `validate-data` 会同时严格验证基础行情与 PIT 侧车；任一身份、完整性或覆盖率不合格都会失败。

## 7. 验收边界

本阶段自动化测试覆盖：公告当日不可见、周末映射、修订时序、旧报告期晚修订、未来值扰动隔离、伪造可用日拒绝、估值时效、组合快照、分区封存、数据和基础身份篡改、未封存文件、覆盖率、断点分区封存、接口字段差异、旧配置迁移、未来配置拒绝和快照 SHA256。

本阶段没有云端 Tushare Token，因此没有把合成测试描述成真实全市场下载。真实使用还需在具备相应接口权限的本地环境执行 `pit-download` 和 `pit-verify`，并归档两份缓存及其 manifest。

## 8. 已知限制与下一阶段入口

- Tushare 上游数据可能修订；封存保证“这份缓存不变”，不保证不同下载日期取得相同上游历史。
- 单一 `announcement_date + lag` 是保守日级模型，不能重建公告的精确时刻、盘中可见性或交易所逐笔信息传播。
- 本阶段未覆盖一致预期、分析师预测、新闻、宏观数据的原始发布日期与修订历史。
- 财务报表的累计值、单季度值、同比增长、TTM 和银行/非银口径尚未做因子层语义处理。
- PIT 数据尚未接入生产 `MultiFactorStrategy`，所以 v1.6 的历史收益不会因为本阶段代码而提高或改变。

下一阶段应在此契约上建立基本面/估值因子注册表、逐信号日覆盖与暴露诊断、IC/分组收益、消融和滚动验证，再决定是否存在值得进入候选组合的增量 Alpha。
